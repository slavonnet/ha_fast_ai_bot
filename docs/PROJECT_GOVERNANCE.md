# Project Governance

## 1. Цель документа

Этот документ фиксирует единые правила работы над проектом:

- как учитывать этапы и подзадачи;
- как вести GitHub issues/PR;
- как не допускать потерь задач между этапами;
- как работать через TDD с UserStory-тестами;
- как улучшать AGENTS.md без раздувания.

## 2. Базовая модель учета работ

Принята модель:

1. **Milestone = Stage**  
   Каждый этап (`Stage`) ведется отдельным GitHub Milestone.
2. **Stage Meta Issue**  
   На каждый этап создается 1 главная issue со всеми:
   - целями;
   - входными/выходными данными;
   - ограничениями;
   - чеклистом подзадач.
3. **Subtask Issues**  
   Каждая подзадача этапа — отдельная issue, связанная с Meta Issue.
4. **Tech Debt Issues**  
   Любой обнаруженный техдолг — отдельная issue с ссылкой:
   - на исходную подзадачу/PR;
   - на влияние и риск.

## 3. Правила по PR и качеству

Каждый PR обязан проходить quality gates:

- `100%` test coverage;
- `100%` docs coverage;
- dependency vulnerability audit (high/critical -> fail).

Если даже один gate не пройден — PR не merge.

## 4. Правило "ничего не упущено"

Для каждого Stage перед закрытием Meta Issue выполняется проверка:

1. Все planned подзадачи этапа имеют один из статусов:
   - `Done`, или
   - явно перенесены в отдельный Stage с ссылкой.
2. Нет "сиротских" задач:
   - в текущем этапе не должно остаться реализации без issue;
   - в следующих этапах не должно отсутствовать продолжение незавершенных протоколов/фич.
3. Если реализована часть функционала (пример: 2 из 5 протоколов):
   - должны существовать отдельные issues на оставшиеся 3;
   - в Meta Issue должен быть комментарий с обоснованием переноса.

Контроль выполняется чеклистом в Meta Issue и PR.

## 5. TDD policy (обязательная)

Разработка ведется через тесты:

1. На каждую подзадачу создается от `5` до `10` UserStory.
2. Каждая UserStory описывается как тест верхнего уровня.
3. Внутри UserStory допускаются дочерние тесты (sub-scenarios):
   - happy path;
   - edge cases;
   - error/fallback.
4. Порядок работы:
   - написать failing tests;
   - реализовать минимальный код;
   - довести до green;
   - рефакторинг без деградации покрытия.

## 6. Работа с агентами (параллельная разработка)

1. В каждой подзадаче можно запускать параллельных агентов.
2. Каждому агенту — отдельная узкая цель и checklist.
3. Коммуникация в задачах/комментариях должна быть:
   - readonly по смыслу для внешних участников (без приватных данных);
   - с фиксацией итогов, а не внутренней "черновой" переписки.
4. После закрытия подзадачи:
   - короткая ретроспектива ошибок;
   - предложение по оптимизации AGENTS.md.

## 7. Правило оптимизации AGENTS.md

После каждой закрытой задачи:

1. Ответить на 3 вопроса:
   - Что мешало агенту выполнить задачу с первого раза?
   - Какую инструкцию можно упростить/объединить?
   - Что можно автоматизировать вместо текстового правила?
2. Изменения в AGENTS.md делать только если они:
   - уменьшают двусмысленность;
   - не дублируют существующие правила;
   - повышают воспроизводимость процесса.

## 8. Определение завершения этапа (Stage DoD)

Этап может быть закрыт, только если:

- выполнены все subtask issues или оформлен перенос;
- quality gates в PR зеленые;
- обновлены документы этапа и changelog в Meta Issue;
- технический долг зафиксирован отдельными issues;
- есть комментарий "stage closure review" с проверкой, что ничего не пропущено.
